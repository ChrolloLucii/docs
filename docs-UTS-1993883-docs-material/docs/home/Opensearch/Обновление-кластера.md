# Обновление кластера OpenSearch

Инструкция по пошаговому обновлению и перезагрузке кластера OpenSearch.

## Подготовка к обновлению

- Выполнить резервные копии конфигураций.
- Сделать снапшоты состояния кластера и индексов.

!!! warning "Важно"
    Узлы OpenSearch не могут быть понижены. Для отката потребуется новая установка и восстановление из снапшота. Сохраните снапшоты в удаленном репозитории перед началом процедуры.

## Создание снапшотов

Для создания снапшота всех индексов:

```http
PUT _snapshot/os-fs-repository-rtest/all_index_snapshot?wait_for_completion=false
{
  "indices": "*",
  "ignore_unavailable": true,
  "include_global_state": true
}
```

Где:

- `os-fs-repository-rtest` — имя репозитория снапшотов (проверка: `GET _snapshot`).
- `all_index_snapshot` — имя снапшота.
- `indices` — маска индексов.

Проверка статуса:

```http
GET _snapshot/os-fs-repository-rtest/all_index_snapshot/_status
```

Поле `state` указывает статус. Создание снапшотов также можно настроить через ISM.

## Пошаговое обновление

1. Проверьте состояние кластера:

```http
GET /_cluster/health?pretty
```

Пример ответа:

```json
{
  "cluster_name": "opensearch-dev-cluster",
  "status": "green",
  "timed_out": false,
  "number_of_nodes": 4,
  "number_of_data_nodes": 4,
  "active_primary_shards": 1,
  "active_shards": 4,
  "relocating_shards": 0,
  "initializing_shards": 0,
  "unassigned_shards": 0,
  "delayed_unassigned_shards": 0,
  "number_of_pending_tasks": 0,
  "number_of_in_flight_fetch": 0,
  "task_max_waiting_in_queue_millis": 0,
  "active_shards_percent_as_number": 100.0
}
```

2. Отключите репликацию шардов:

```http
PUT /_cluster/settings?pretty
{
  "persistent": {
    "cluster.routing.allocation.enable": "primaries"
  }
}
```

Пример ответа:

```json
{
  "acknowledged": true,
  "persistent": {
    "cluster": {
      "routing": {
        "allocation": {
          "enable": "primaries"
        }
      }
    }
  },
  "transient": {}
}
```

3. Выполните flush:

```http
POST /_flush?pretty
```

Пример ответа:

```json
{
  "_shards": {
    "total": 4,
    "successful": 4,
    "failed": 0
  }
}
```

4. Определите порядок обновления узлов:

- Data nodes
- Ingest/ML/Coordinating nodes
- Cluster manager nodes

!!! warning "Важно"
    Узлы-менеджеры кластера обновляются последними. Узлы OpenSearch могут подключаться к более старому менеджеру, но не к более новому.

5. Определите текущий узел-менеджер:

```http
GET /_cat/nodes?v&h=name,version,node.role,master
```

Пример вывода:

```text
name        version  node.role  master
os-node-01  7.10.2   dimr       -
os-node-04  7.10.2   dimr       -
os-node-03  7.10.2   dimr       -
os-node-02  7.10.2   dimr       *
```

6. Остановите узел, который обновляете.
7. Подтвердите, что узел вышел из кластера:

```http
GET /_cat/nodes?v&h=name,version,node.role,master
```

8. Обновите пакет OpenSearch на новую версию и запустите службу.
9. Убедитесь, что узел вернулся в кластер:

```http
GET /_cat/nodes?v&h=name,version,node.role,master
```

10. Включите репликацию шардов:

```http
PUT /_cluster/settings?pretty
{
  "persistent": {
    "cluster.routing.allocation.enable": "all"
  }
}
```

11. Убедитесь, что кластер здоров:

```http
GET /_cluster/health?pretty
```

12. Повторите шаги 2–11 для каждого узла (менеджер — последним). После замены всех узлов проверьте версии:

```http
GET /_cat/nodes?v&h=name,version,node.role,master
```

## Пошаговая перезагрузка

Пошаговая перезагрузка следует той же процедуре, что и обновление, но без замены бинарного файла. Узлы перезагружаются по одному для применения конфигураций/сертификатов без простоя.

1. Проверьте состояние кластера (шаг 1 обновления).
2. Отключите перераспределение шардов (шаг 2 обновления).
3. Выполните flush (шаг 3 обновления).
4. Определите следующий узел, менеджер перезагружайте последним.
5. Проверьте текущего менеджера (`_cat/nodes`).
6. Остановите узел, не удаляя данные.
7. Убедитесь, что узел вышел из кластера.
8. Запустите узел с тем же бинарником/конфигурацией.
9. Проверьте, что узел вернулся в кластер.
10. Включите перераспределение шардов.
11. Убедитесь, что кластер в состоянии `green`.
12. Повторите для остальных узлов.

Сохранение кворума и последовательная перезагрузка обеспечивают нулевое время простоя и непрерывность данных.
